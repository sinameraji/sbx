Memory floor — lower bound (Hetzner 5950X bare metal, 8 vCPU pinned CCD0)

The workload dies at TYPECHECK below a memory floor. Confirmed lower bound so far:

  mem_mb  cpus  result
  4096    8     OOM
  6144    8     OOM
  8192    8     OOM

=> ≤ 8 GiB always OOMs at typecheck. OOM is memory-deterministic (the guest has a fixed
memoryMB; contention cannot manufacture one), so these binary results are valid even
though these particular runs' *timings* were contended by a concurrent diagnostic — see
`run-5c49059-contention-and-hang.txt`. The published 32 vCPU row ran at 32 GiB, well
above the floor.

The floor is a BAND, wider than a first pass suggested. Peak working set spans
10.7–15.8 GiB across identical runs, so every rung from 12 to 16 GiB can flip. The
committed per-run record:

  rung    clean ladder (ladder-runs/)   earlier session (session-1547Z-runs/)
  12 GiB  OOM/PASS/OOM  (1/3)           OOM/OOM/OOM  (0/3, peaks 11.71/11.71/11.65)
  14 GiB  PASS/PASS/PASS(3/3)           (not run this session)
  16 GiB  PASS/PASS/PASS(3/3)           OOM/PASS/PASS(2/3, OOM rep peak 15.65 GiB)

An even earlier run saw 12 GiB pass 2/3, but its per-run raws were overwritten before
we adopted publish-everything — that count is disclosed as recollection, not evidence.
Only ≥ 24 GiB has never OOM'd in any session (max observed peak 15.75 GiB). Full per-run
stdout for both sessions is committed under ladder-runs/ and session-1547Z-runs/.
