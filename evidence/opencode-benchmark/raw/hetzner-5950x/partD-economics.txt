Part D — economics (v2, measured-only). Hetzner 5950X Server Auction, €0.36/hr,
125 GiB RAM, no swap, 3.4 TB NVMe root (2nd NVMe unused).

Every number below is either (a) MEASURED — traceable to a raw file in this
directory, or (b) ARITHMETIC — plain division over measured per-unit numbers,
labeled as such. Nothing is modeled or extrapolated.

MEASURED — at rest (paused sandbox):
- Host RAM per paused sandbox ≈ 6.6 MB. 90 paused 2 GiB sandboxes moved host
  MemAvailable 127140 -> 126542 MB = 598 MB total (partC-work-spike.txt).
  Pause kills the VMM; state lives on disk, not RAM.
- Disk per paused 2 GiB sandbox = 2.22 GB. 80 paused = 177617 MB total
  (partC-spike.txt).
- Disk per paused 24 GiB sandbox = 27.8 GB: memory snapshot 24577 MB +
  workspace image 3217 MB (partB-pauseresume.txt). Snapshots are Full, not
  diff — ratio under zstd is a separate line item (B7).

MEASURED — waking them (thundering herd):
- 80 paused woken AT ONCE: 80/80 resumed, 0 failed, 0 OOM; whole herd 381 ms;
  per-resume p50 265 / p95 350 / p99 357 ms (partC-spike.txt).
- Uncontended resume p50 65 ms over 50 cycles (partB-pauseresume.txt).
- Resume is lazy-paged (file-backed mmap): a resumed-but-idle guest costs its
  WORKING SET, not its allocation — host MemAvailable barely moved on wake.

MEASURED — active-concurrent (the number that was previously a guess):
- 81 concurrently-WORKING 2 GiB guests, each actively touching 1.5 GiB of
  tmpfs, ran on the 125 GiB host before MemAvailable bottomed at 1408 MB —
  near exhaustion, 0 OOM kills, and NO admission refusal (partC-work-spike.txt).
  That last part is the admission-gap finding (Part C): the host over-commits
  on the resume path rather than refusing.

ARITHMETIC (division over the measured per-unit numbers, nothing else):
- Paused density if you fill the 3.4 TB root NVMe: ~1530 × 2 GiB sandboxes,
  or ~122 × 24 GiB. The 2nd unused NVMe would double both.
- €/paused-sandbox/hr at that fill: €0.36 / 1530 ≈ €0.00024 (2 GiB);
  €0.36 / 122 ≈ €0.0030 (24 GiB).

NOT MEASURED (honest gaps):
- The paused-density limit itself (we never filled the disk).
- The OOM breaking point past 81 workers (we reached near-exhaustion and
  stopped; 0 kills at 81).
- An active-concurrent ceiling for HEAVY workloads. The OpenCode typecheck
  needs the ~14 GiB floor (demand-curve-ladder.tsv), so a 125 GiB host bounds
  it at ≤8 concurrent — but that is a bound from the floor, not a measured
  wall, and is flagged as such.

Workload footprint (from the benchmark's own BENCH_DISK): after_install
≈ 3.2 GB node_modules.
